# ChromBERT 预训练数据集与规模分析 (Promoter vs. Whole Genome)

在深入研究原始论文、源码库及挂载的数据集之后，我们发现 ChromBERT 实际上包含**两个完全不同规模和生物学目标的预训练任务**。本文档记录了两者的差异、数据大小及相应的工程挑战。

## 1. 启动子数据集 (Promoter Data) —— 轻量与功能聚焦
这是目前我们在基础复现任务中（以及 `FlashChromBert` 项目中）主要处理的版本。
* **数据来源**：ROADMAP (包含 127 种细胞类型)
* **染色质状态标准**：**15 态** (15-state annotations)
* **覆盖范围**：仅提取了基因转录起始位点 (TSS) 上下游的区域（$-2\text{ kb}$ 到 $+4\text{ kb}$），即启动子功能区。
* **词表配置**：对应代码中的 `bert-config-4`。
* **物理大小**：解压后的文本文件 (`pretrain_4mer_all.txt`) 大小约为 **2.8 GB**，包含 2,685,105 行样本。
* **工程特征**：数据量适中，可以被现代服务器一次性完整加载到内存 (RAM) 中。

## 2. 全基因组数据集 (Whole Genome Data) —— 庞大与全局拓扑
这是论文中提到的另一种更具挑战性的预训练场景，旨在让模型学习长距离调控和 TAD（拓扑相关结构域）等全局边界信息。
* **数据来源**：IHEC
* **染色质状态标准**：**18 态** (18-state annotations)
* **覆盖范围**：覆盖人类的整条染色体（包含大量非编码区、异染色质等荒漠区域）。
* **词表配置**：对应代码中的 `bert-config-18-4`。
* **物理大小**：解压后的文本文件 (`pretrain_genome_all.txt`) 大小高达 **41.5 GB**。
* **工程特征**：
    * **极高的压缩率**：由于染色体状态序列（如连续的背景静息状态 R）具有极高的重复度，这 41.5 GB 的庞大文本在压缩后，可以和所有其他产物一起塞进区区 3.2 GB 的 `ChromBERT_ver01_light.zip` 压缩包中。
    * **内存溢出困境 (OOM)**：由于普通服务器无法一次性将 41 GB 文本载入内存，原作者在 `pretraining_loop.sh` 中被迫采用了**切块循环 (Split chunks)** 策略：将 41GB 切割成若干小文件，写一个 bash `for` 循环，每次加载一块并训练 5000 步，通过 Perplexity (困惑度) 阈值控制早停。

## 3. 核心差异总结与后续建议

| 维度 | Promoter 预训练 | Whole Genome 预训练 |
| :--- | :--- | :--- |
| **生物学意义** | 基因开关的局部调控模式 | 全局染色质三维拓扑与长距离依赖 |
| **文本大小 (4-mer)** | **~2.8 GB** | **~41.5 GB** |
| **状态字母表** | 15 种状态 | 18 种状态 |
| **训练策略** | 可全量加载，传统 DDP 训练 | 切块流式训练 (或需改用 WebDataset) |

**工程建议**：
将来如果要执行全基因组的训练任务（41.5 GB），强烈建议**不要**沿用原作者那种“打断式的 for 循环切块”方案。对于现代化训练，应该利用 PyTorch Lightning 的 `IterableDataset`，或基于 HuggingFace `datasets` 库的内存映射 (memory-mapping / memmap) 机制，实现真正的流式读取。这样既能打破内存瓶颈，又能保证数据分布的随机性和模型训练的连续性。
